  CISPO是一種強化學習算法，旨在提升大模型訓練過程的穩定性，加快其收斂—大模型達到最佳性能的點。
  傳統上，大模型採用「裁剪」技術，通過去除在模型訓練過程中引發不穩定性的token來穩定模型訓練。
  然而，這個過程可能在無意中丟棄有價值的學習信號。
  CISPO的創新之處在於「裁剪」重要性抽樣權值（表示數據重要性的分數）。
  這確保所有數據，即使是次要元素，都有助於核心學習機制。
  這項技術防止了關鍵學習信號的丟失，從而培育出一個更可靠的大模型訓練過程。
  CISPO的效率在於其能夠實現與GRPO和DAPO等現有算法相當的性能，
  而在現實世界測試中，它只需要約一半的訓練迭代（或模型學習周期）。

- 向豆包学习
  「是什么 - 测什么 - 怎么用」
  定位、评估逻辑、使用场景和实操方法
  传统代码基准（如 HumanEval）：让模型 “写一个函数”（孤立任务）；
  SWE-bench：让模型 “修复 Django 项目中某个真实 bug”“给 Redis 加一个指定功能”（真实工程场景）。
  
  核心设计理念
  1. 理解问题：看懂 issue 描述、代码上下文、报错日志；
  2. 定位根因：在数万行代码中找到 bug 所在的文件 / 行数；
  3. 编写修复代码：修改代码、补全测试、保证兼容性；
  4. 验证修复：确保修改后项目能运行、测试用例通过。
  
  数据集构成
  * 覆盖项目：50+ 主流开源项目（Django、Redis、Scikit-learn、PyTorch、TensorFlow 等）；
  * 任务类型：10k+ 真实 issue，包含 bug 修复、功能开发、性能优化、兼容性调整；
  * 数据来源：从 GitHub 开源项目的已关闭 issue 中筛选，确保任务有明确的 “问题描述 + 正确修复方案”；
  * 难度分级：
    - Easy：简单语法 bug、单文件修改；
    - Medium：多文件联动修改、逻辑 bug；
    - Hard：架构级调整、跨模块兼容、性能优化。 

  核心评估指标
  * 解决率（Solve Rate）模型能完整修复 / 实现的任务比例（核心指标）
  * 定位准确率（Localization Accuracy） 模型能找到需要修改的文件 / 行数的比例
  * 测试通过率（Test Pass Rate） 修复后项目单元测试、集成测试的通过率
  * 代码兼容性（Compatibility） 修改后的代码是否兼容项目现有逻辑、无新 bug
  * 迭代次数（Iteration Count） 模型完成修复所需的尝试次数（越少越好）

  与传统代码基准的核心差异
  SWE-bench 真实软件工程能力（修 bug / 开发功能） GitHub 真实 issue
  HumanEval HumanEval 人工编写的简单函数
  MBPP 基础编程任务 人工编写的小任务
  LiveCodeBench 实时交互式调试 人工设计的调试场景

  TAU-bench（τ-bench）是评估AI 智能体在真实业务场景中，
  与用户、工具、规则进行多轮交互能力的权威基准，
  核心测 “工具调用 + 规则遵守 + 多轮对话 + 状态一致性”，是当前 Agent 能力评测的标杆。

  TAU-bench 要求智能体在多轮对话中，
  一边理解用户意图、一边调用 API、一边严格遵守业务规则，最终完成复杂业务目标。
  对话理解、工具调用、规则遵循、状态一致性、多轮推理。

# 每日提示词

现在请对生成的稿件进行最后的交付测试。
排查长句：找出所有超过20个字的句子并进行拆解，确保主讲人换气顺畅。
口语翻译：将所有书面语（如“致力于、由此可见”）替换为日常交流语言（如“咱们死磕、其实大家看来”）。
模拟质检：请模拟一位挑剔的听众，指出文中哪一段话最容易让人走神，并给出修改建议。
朗读校对：加入3处引导互动的停顿提醒，确保整篇稿子即便不看稿也能顺着逻辑讲下来。
