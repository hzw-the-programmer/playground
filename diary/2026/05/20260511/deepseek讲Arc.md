详解 Arc<T> 实现了 Send 和 Sync（当 T 满足相应条件时）。

---

我们接着上次的话题，详细拆解一下 `Arc<T>` 对 `Send` 和 `Sync` 的实现条件。核心结论是：

```rust
impl<T: ?Sized + Send + Sync> Send for Arc<T> {}
impl<T: ?Sized + Send + Sync> Sync for Arc<T> {}
```

也就是说，`Arc<T>` 同时实现 `Send` 和 `Sync`，当且仅当 `T: Send + Sync`。
下面逐步解释为什么需要这两个条件缺一不可，以及它们各自保护了什么。

---

# 1. 快速回顾：`Send` 与 `Sync` 在说什么

- `Send`：类型的所有权可以安全地转移到另一个线程。
- `Sync`：类型的共享引用（&Self）可以安全地在多个线程间同时存在并被使用。

注意：`Sync` 说的是“共享引用在多线程下安全”，不是所有权转移。如果一个类型 `T: Sync`，那么 `&T` 就是 `Send`。

---

# 2. `Arc<T>` 的内部结构本质

`Arc<T>` 可以看作一个指向堆上数据的指针，堆上数据包含两部分：

- 原子引用计数（强计数、弱计数）
- 实际数据 `T`

当你 `Arc::clone` 时，会复制这个指针，然后对原子计数进行一次原子加法；丢弃 Arc 时原子减法，减到零就释放整个堆内存。

原子操作本身保证了计数器的线程安全，但这和 `T` 的线程安全没有直接关系——`T` 是通过多个 `Arc` 指针被共享的。

---

# 3. 为什么 `Arc<T>: Send` 需要 `T: Send + Sync`

`Arc<T>` 要变成 `Send`，意味着可以把一个 `Arc<T>` 的所有权安全地发送给另一个线程。
这看起来很简单：只要 `T` 能被发送（`T: Send`）不就行了吗？
不，还需要 `T: Sync`。 原因在于：你发送一个 `Arc<T>` 到新线程后，原线程可能还持有从这个 `Arc` 克隆出的其他 `Arc<T>`。示例如下：

```rust
let arc1 = Arc::new(data);
let arc2 = Arc::clone(&arc1);

// 线程A持有 arc1，线程B拿到 arc2
thread::spawn(move || {
    // 线程B中使用 arc2
});
// 线程A中仍在使用 arc1
```

此时，`data` 同时被线程 A 和线程 B 通过共享引用（`&T`）访问。
如果 `T` 只是 `Send` 但非 `Sync`，则多个 `&T` 同时存在在多线程中是不安全的（因为 `T` 的内部可变性可能不保护并发，比如 `RefCell`、`Cell`）。因此，为了能让一个 `Arc` 的所有权安全移动到另一个线程，就必须保证即使以后出现多个 `Arc` 指向同一 `T` 的情形，这个共享也是安全的——即要求 `T: Sync`。

所以 `Arc<T>: Send` 的条件是 `T: Send + Sync`，而不仅仅是 `T: Send`。

---

# 4. 为什么 `Arc<T>: Sync` 也需要 `T: Send + Sync`

`Arc<T>` 要成为 `Sync`，意味着可以在多个线程中安全地共享 `&Arc<T>`，即可以把这个共享引用发给别的线程使用。

通过 `&Arc<T>` 我们能做什么？最典型的就是 `Arc::clone`，因为它只需要一个共享引用：

```rust
let a: &Arc<T> = ...;
let clone = Arc::clone(a);   // 只需要 &self
```

有了 `&Arc<T>`，任何线程都能克隆出一个 `Arc<T>`，然后把这个克隆体（所有权）发送到别的线程（这就要求 `Arc<T>: Send`）。
而 `Arc<T>: Send` 我们已经知道需要 `T: Send + Sync`。同时由于这个过程中会制造多个指向同一 `T` 的 `Arc`，`T` 本身也必须是 `Sync`。
所以 `Arc<T>: Sync` 的条件最终与 `Send` 完全一致：`T: Send + Sync`。

简单记忆：`Arc` 天生就允许把一份数据共享到多个线程，因此里面的 `T` 必须既能“发送”（所有权转移时）又能“共享引用安全”（多线程 `&T` 同时存在时）。

---

# 5. 如果 T 不满足条件会怎样？

- `T` 是 `!Send`（比如 `Rc`、`*mut T`、裸指针相关的非 `Send` 包装等）：
那么 `Arc<T>` 既不是 `Send` 也不是 `Sync`，根本不能用 `thread::spawn` 传递所有权，也无法在线程间共享 `&Arc<T>`。

- `T` 是 `Send` 但 `!Sync`（典型：`RefCell<T>`、`Cell<T>`）：
这些类型本身可以安全地转移所有权到另一个线程，但不能让多个线程同时持有其共享引用（因为其内部可变性使用非原子操作）。
此时 `Arc<RefCell<T>>` 既不是 `Send` 也不是 `Sync`。你会收到类似错误：`Arc<RefCell<T>>` cannot be sent between threads safely。

常见解决方案：

- 用 `Mutex<T>`、`RwLock<T>` 包装，它们通常是 `Sync`（当 `T: Send` 时），于是 `Arc<Mutex<T>>` 就能实现 `Send` + `Sync`。
- 或者用 Atomic* 等原生同步原语。

---

# 6. 一个例外：`Arc::get_mut` 与独占访问

当 `Arc` 的强引用计数为 1 时，我们可以获取 `&mut T` 而不需要锁。这时虽然有 `Arc`，但逻辑上等效于独占所有权。但 trait 实现是编译期固定的，与运行时计数无关，所以不改变 Send/Sync 的条件。

---

# 总结一下：

`Arc<T>` 内部原子计数本身是多线程安全的，但它的职责是共享 `T`。
为了让共享同一个 `T` 的所有 `Arc` 可在多线程间随意克隆、发送和访问，`T` 必须保证：

- 能发送到其他线程（`Send`）
- 共享引用在多线程下安全（`Sync`）

因此标准库让 `Arc<T>` 同时实现 `Send` 和 `Sync`，当且仅当 `T: Send + Sync`。
这就是“当 T 满足相应条件时”的准确含义。
