Rust 的 async/await 语法是零成本抽象的典范。编译器会在编译期将每个异步块或异步函数转换为一个状态机（state machine），并为其生成一个实现了 Future trait 的匿名类型。下面我们从原理到细节，完整剖析这一过程。

# 1. 为什么需要状态机？

异步函数可能因为 .await 而暂停多次。暂停时，所有跨 .await 点仍有效的局部变量、以及当前执行到哪一步，都必须被保存起来，以便将来恢复执行。状态机就是为了精确记录这些信息。

```rust
async fn simple(x: i32) -> i32 {
    let a = 1;
    let b = read_file().await;  // 暂停点1
    let c = a + b;
    process(c).await;           // 暂停点2
    c + x
}
```

函数会在 read_file().await 和 process(c).await 两处可能暂停。每次暂停时，a、b、c 等变量需要保留，同时要记住“下次恢复时从哪里开始”。

# 2. 编译器转换流程

1. 解析 AST：async 块被识别为 AsyncBlock。

2. HIR → MIR：编译器将其降级为一个 generator（生成器）。Rust 的 async/await 就是 generator 的语法糖，generator 的 yield 对应 .await。

3. 生成状态机：MIR 层为这个 generator 创建内部状态类型，并将所有跨 yield 点的变量提升为结构体字段。

4. 实现 Future：基于 generator 的 resume 自动实现 Future::poll。

# 3. 状态机的内部形态

尽管早期常以 enum 来讲解状态机，但现代 Rust 编译器实际上使用 struct + 整数状态码 的方式实现。编译器会生成一个独有的匿名结构体，包含：

- 状态码：一个 usize 字段，指示当前执行到哪个暂停点之前（0 表示起点，1 表示第一个 .await 之后，2 表示第二个 .await 之后等）。

- 跨 .await 的局部变量：所有在下一次暂停后还会被用到的变量。

- 当前正在执行的子 Future：例如 read_file() 返回的 Future，在等待它时也需要保存在状态机内。

以开头代码为例，编译器生成的结构体大致为：

```rust
struct SimpleAsyncStateMachine {
    state: usize,          // 当前状态码
    x: i32,                // 参数 x（始终有效）
    a: i32,                // 跨暂停点有效
    b: i32,                // 第二个暂停点后使用
    c: i32,                // 第三个暂停点后使用
    read_fut: ReadFuture,  // 挂起的 read_file() future
    process_fut: ProcessFuture, // 挂起的 process() future
}
```

# 4. poll 方法的执行逻辑

生成的 Future::poll 方法内部是一个 loop + match self.state 的结构：

```rust
impl Future for SimpleAsyncStateMachine {
    type Output = i32;
    fn poll(mut self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<i32> {
        loop {
            match self.state {
                // 状态 0：起始，执行 a = 1，然后启动 read_file()，进入状态1
                0 => {
                    self.a = 1;
                    self.read_fut = read_file();
                    self.state = 1;
                    // 继续执行，立刻尝试推进 read_fut
                }
                // 状态 1：等待 read_file() 完成
                1 => {
                    match self.read_fut.as_mut().poll(cx) {
                        Poll::Ready(val) => {
                            self.b = val;
                            self.c = self.a + self.b;
                            self.process_fut = process(self.c);
                            self.state = 2;
                            // 继续执行，尝试推进 process_fut
                        }
                        Poll::Pending => return Poll::Pending,
                    }
                }
                // 状态 2：等待 process() 完成
                2 => {
                    match self.process_fut.as_mut().poll(cx) {
                        Poll::Ready(()) => {
                            self.state = 3;
                            // 完成，返回最终结果
                        }
                        Poll::Pending => return Poll::Pending,
                    }
                }
                // 状态 3：完成
                3 => {
                    return Poll::Ready(self.c + self.x);
                }
                _ => unreachable!(),
            }
        }
    }
}
```

关键点：

- 整个状态机是一个 loop，因为 .await 之后的代码可能会在同一个 poll 调用中立即执行（如果子 future 已就绪），这样就减少了调度开销。

- 只有在真正遇到 Poll::Pending 时才返回，同时将 Context 中的 waker 注册给子 future，以便将来唤醒。

- 状态码在每次跨越 .await 后自增，确保下一次 poll 时直接跳转到正确的位置。

# 5. 哪些变量会被保存？

只有 跨 .await 点仍然活跃 的局部变量才会提升为状态机的字段。编译器会进行活跃性分析：

```rust
async fn demo() {
    let x = 1;
    let y = 2;
    let z = read().await; // 使用 x, y
    let w = x + y + z;    // 使用 x, y, z
    // 此后不再使用 x, y, z
    write(w).await;
}
```

- x、y、z 跨越了第一个 .await，会被保存。

- 在第二个 .await 之后，x、y、z、w 都无效了，因此它们不会占用状态机在第二个暂停点的空间（但在实际实现中，为了简化，编译器可能仍会在整个状态机中保留这些字段，但可以重用内存）。

# 6. Pin 的必要性

由于跨 .await 的变量被保存在状态机内部，而且某个局部变量可能是对另一个局部变量的引用（自引用结构），这种引用关系在状态机移动后就会失效。但 Future 本身是可以移动的（例如传递给 spawn）。Pin 就是为了解决这个问题：

- poll 函数签名要求 self: Pin<&mut Self>，这保证了一旦状态机被 Pin 住，它就不会在内存中被移动。

- 编译器生成的代码会使用 Pin::get_mut 或投射来安全地访问字段，防止移动。

- 自引用常见于：let buf = [0u8; 1024]; let reader = read_into(&mut buf);，这里 reader 内部持有对 buf 的引用，如果状态机移动，引用就失效了。

# 7. 编译器优化

- 状态码合并：如果状态机很简单，状态码可能被编码为更小的整数，甚至被优化掉。

- 字段复用：编译器会用类似 union 的方式在不同状态中重叠存储那些生命周期不重叠的变量，减小状态机大小。

- 内联：当异步块非常小时，编译器可能将其完全内联到父状态机中。

- 零尺寸优化：如果异步块没有跨 .await 的变量，状态机可能只是一个零大小的 struct，poll 直接同步执行。

# 9. 总结

- Rust 的 async 块被编译器转换为一个实现了 Future 的匿名结构体，内部包含状态码 + 跨暂停点变量 + 子 Future。

- poll 方法通过匹配状态码来恢复执行，遇到 Pending 时暂停并返回，待唤醒后再次进入 poll 从上次中断的状态继续。

- 这种生成的状态机保证了零成本：执行效率与手写状态机相同，没有额外的装箱或调度开销（除非你手动 Box::pin 或 spawn）。

- Pin 机制保障了自引用安全，让 async 块可以自由地保存引用到内部变量。

这种设计是 Rust 异步生态高性能的关键，它让你能用同步的书写方式，享受异步运行时的极致效能。
