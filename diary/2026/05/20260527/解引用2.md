`val[0]` 展开时一定会出现 `*` 号，这是编译器对索引表达式的强制解糖规则，与后续跟什么操作无关。

`val[0].clone()` 的完整展开过程如下：

---

# 第一步：索引表达式解糖

```rust
val[0].clone()
// 先展开为：
(*val.index(0)).clone()
```

无论 `val[0]` 后面跟着什么（赋值、方法调用、传参……），编译器都会先将 `val[0]` 替换为 `*val.index(0)`。

此时 `val.index(0)` 返回 `&T`，`*` 解引用得到一个 `T` 类型的占位表达式（place expression）。

---

# 第二步：方法调用自动引用

`.clone()` 方法的签名是 `fn clone(&self) -> Self`，它需要 `&self`。

但此刻调用者是 `*val.index(0)`，其类型是 `T`（不是引用）。

Rust 的方法查找机制发现 `T` 没有接受 `self` 的 `clone`，于是自动为调用者添加一层引用，转换为：

```rust
(&(*val.index(0))).clone()
// 即
(&*val.index(0)).clone()
```

---

# 第三步：引用-解引用抵消

`val.index(0)` 已经返回 `&T`，`*` 解引用再 `&` 取引用，两者抵消，最终等价于：

```rust
val.index(0).clone()
```

这就是为什么即使 `T` 不是 `Copy`，`val[0].clone()` 依然能编译通过——实际克隆的是引用指向的值，而不会把所有权从容器中移出。

---

# 总结

- 展开必有 `*`：`val[0]` → `*val.index(0)`，这一步是硬性规定。

- 自动引用消除影响：因为 `.clone()` 需要 `&self`，编译器对 `*val.index(0)` 重新取引用，导致 `*` 与 `&` 抵消，最终效果等同于直接在引用上调用 `clone`。

- 对 `Copy` 类型同理：`*val.index(0)` 会复制出一个值，然后在该值上调用 `clone()`，同样正确。

所以，`*` 号在解糖过程中实际存在，但在调用 `clone` 的最终形态中它被自动引用的 `&` 抵消了，从而让我们无需手动处理引用。 这也是 Rust 索引表达式设计精巧之处——它总能兼容值操作与方法调用。
