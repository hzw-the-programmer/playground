为什么不用PhantomData

---

你这个问题问得很好，把这两种看似都能“占坑”的写法放在一起对比，才能看清楚各自的分工。

简单来说：**用 `()` 还是 `PhantomData`，取决于这个字段是“纯粹的私有占位符”还是“给编译器看的幽灵数据”**。

如果结构体**没有泛型参数、不涉及生命周期、也不需要告诉编译器任何所有权/型变信息**，那么 `_p: ()` 是更直接的选择。原因如下：

---

# 1. 语义清晰，意图明确

`()` 就是一个零大小的空元组，用它做私有字段，表达的意思非常单纯：
**“我就是不存数据，纯粹为了防止你在外部用字面量构造我。”**

而 `PhantomData<T>` 本身是一个**标记类型**，它的职责是向编译器传递“假装拥有 T”的信息，常用于：

- 关联生命周期（`PhantomData<&'a ()>`）
- 声明所有权（`PhantomData<T>` 表示“逻辑上拥有 `T`”）
- 控制型变（variance）

如果写 `_p: PhantomData<()>`，虽然也能禁止外部构造（因为字段私有），但会引入一个多余的语义：**“这个结构体假装拥有 `()`”**。这会让阅读代码的人分心，误以为将来会替换成有意义的类型参数。而用 `()` 就不会带来这种困惑。

---

# 2. 简洁，少引用一个标准库类型

`()` 是原语，零开销，无需 `use std::marker::PhantomData;`。对于这种极简的令牌类型，很多人偏好更短的写法。

---

# 3. 构造控制的核心是字段可见性，而不是字段类型

你可能会想：“`PhantomData` 不也能防止外部构造吗？” 没错，**但真正阻止外部构造的是 _p 这个字段是私有的**，跟它的类型是 `()` 还是 `PhantomData<()>` 无关。
外部既不能写 `Identity { _p: () }`（字段不可见），也不能写 `Identity { _p: PhantomData }`（即使 `PhantomData` 是库类型，字段依然不可见）。
既然两个都能做到，为什么不选更简单、语义负载更小的那个呢？

---

# 什么时候就该用 PhantomData ？

如果 `Identity` 将来会变成这样：

```rust
pub struct Identity<'a, T> {
    _p: PhantomData<&'a T>,
}
```

那么 `PhantomData` 就是必需的，因为它要让 `'a` 和 `T` 被编译器认为“已使用”，同时还能控制型变。
但针对一个无参的纯令牌类型，`PhantomData<()>` 反而过度设计了。

---

# 总结
- **当前场景**：`Identity` 是一个零大小、私有构造的权限令牌，没有任何泛型或生命周期——`_p: ()` **是最佳实践**，意图精准，简单高效。
- **如果未来需要幽灵类型**：那就应该把 `PhantomData` 请回来，并带上真正的类型/生命周期参数，而不是写一个无意义的 `PhantomData<()>`。
