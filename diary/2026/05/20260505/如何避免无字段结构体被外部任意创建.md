pub struct Identity {
    _p: (),
}
这么定义有什么好处，解决什么问题

---

在 Rust 中，这种写法通常是为了**创建一个只能在当前模块内部构造、外部无法直接实例化的“零大小类型”（ZST）**。具体好处和解决的问题如下：

# 1. 防止外部代码任意构造实例

如果定义为普通无字段结构体：

```rust
pub struct Identity;
```

那么任何外部代码都可以通过 `Identity` 或 `Identity {}` 直接创建该类型的值，完全不受控。

加上一个**私有的** `_p: ()` 后：

- 字段是私有的，外部无法直接访问
- 结构体字面量构造要求所有字段都可见，因此外部无法用 `Identity { _p: () }` 构造
- 因为 `()` 没有公共构造函数可言，外部也无法在别处凑出这个值传进来
  
于是，**实例的创建权完全被模块作者掌控**，只能通过你提供的公开函数（比如 `new()` 或其他关联函数）来获取实例。这对于以下场景很重要：

- 权限令牌（capability token）：持有该类型实例才能调用某些敏感操作
- 保证内部不变性：所有实例的创建路径都在你的控制之下，可以附加逻辑或校验

# 2. 零大小，无运行时开销

`()` 是零大小类型，所以 `Identity` 本身也是 ZST，不对齐、不占空间。既可以当编译期的“标记”，又完全不会增加内存负担。

# 3. 可组合用于更高级的模式

虽然这个定义本身简单，但常被用于：

- **幽灵类型（PhantomData）的替代 / 补充**：在没有泛型时，作为纯粹的生命周期或类型标记载体（不过没有生命周期参数时更像纯令牌）
- **实现 `Drop` 做 RAII 守卫**：例如 `MutexGuard` 这类类型会私有一个类似字段，防止你在外部 `mem::forget` 或手动构造绕过锁逻辑。通过私有字段，锁的获取只能通过正确的 `lock()` 方法，从而保证解锁行为一定发生。
- **作为条件编译的占位**：在某些 feature 开关下，结构体可能不需存储数据，但又想保持构造控制权，这种写法很合适。

# 总结

好处：完全禁止外部构造，实现“构造权私有化”，同时保持零大小、无开销。
解决的问题：无字段结构体可以被外部任意创建，导致类型失去作为令牌/守卫的控制力。
