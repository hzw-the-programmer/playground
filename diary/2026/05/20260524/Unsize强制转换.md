你观察到的现象——将一个数组的引用 `&[T; N]` 直接传给接受切片 `&[T]` 的函数——在 Rust 里非常常见，比如：

```rust
fn takes_slice(slice: &[i32]) {
    println!("{:?}", slice);
}

let arr: [i32; 3] = [1, 2, 3];
takes_slice(&arr); // &[i32; 3] 传给了 &[i32] 参数，正常编译
```

这背后是 Rust 的一种隐式类型转换，叫做 Unsize 强制转换（unsize coercion）。

核心原因：`[T; N]` 到 `[T]` 是“不定长化”

Rust 的类型系统区分两种大小：

- `Sized` 类型：大小在编译期就知道，比如 `i32`、`[T; N]`。

- `!Sized` (DST) 类型：大小在编译期未知，比如 `[T]`、`dyn Trait`。

`[T]` 这种动态大小的切片不能直接存在于栈上，你只能通过指针（引用、`Box` 等）来操作它。
数组 `[T; N]` 则是完全已知大小的“胖指针”背后那个值。

Unsize 强制转换就是允许将“指向某个 `Sized` 类型的指针”自动变成“指向它对应的 `!Sized` 类型的指针”。具体到数组：

- 指针类型：`&[T; N]`

- 目标类型：`&[T]`

- 这种转换是 `[T; N]` 实现 `Unsize<[T]>` 特性的结果，编译器在需要时会自动插入。

转换发生时机

当函数形参是 `&[T]`，而你传入 `&arr` (类型为 `&[T; N]`) 时，编译器发现类型不匹配，但它知道 `[T; N]` 可以变成 `[T]`，于是自动将 `&arr` 强制转换为一个**包含了数组首地址和长度 N 的胖指针（即真正的切片引用）**。整个过程零开销，不拷贝数据。

其他形式的切片转换

除了隐式强制，还可以用显式的方式达到同样目的，便于理解：

```rust
let arr = [1, 2, 3];
let slice: &[i32] = &arr;          // 隐式强制
let slice2 = &arr[..];             // 索引语法，同样生成 &[i32]
let slice3 = arr.as_slice();       // 方法调用，内部也是 &self[..]
let slice4: &[i32] = arr.as_ref(); // 数组实现了 AsRef<[T]>
```

但它们底层都是同一个原理：**将编译期已知长度的数组，变为运行期携带长度的切片引用**。

与 Deref 的区别

你可能知道 `Vec<T>` 通过实现 `Deref<Target=[T]>` 可以将 `&Vec<T>` 自动解引用成 `&[T]`，但数组并没有实现 `Deref`。数组的转换完全由编译器的 `unsize coercion` 完成，这是语言内建的特殊规则，不需要 trait 支持。

总结

数组引用能作为切片传参，是因为 Rust 允许 &[T; N] 隐式强制转换为 &[T]。这是一种 unsize 强制转换，它利用编译期已知的数组大小，在运行时构造出“指针 + 长度”的胖指针，从而安全、零成本地转换为切片。这个特性是处理数组与切片的核心设计，让代码在静态长度和动态长度间无缝切换。
