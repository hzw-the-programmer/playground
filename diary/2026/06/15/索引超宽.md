在常规的环形缓冲区实现中，索引 `head` 和 `tail` 通常会回绕到 `[0, N-1]` 之间，此时 `head == tail` 既可能表示队列为空，也可能表示队列为满。为了区分，常见的做法是牺牲一个槽位（满时 `tail + 1 == head`），或额外维护一个标志位。无论哪种方式，计算长度都需要进行取模运算，例如 `(tail - head + N) % N`。

而 Tokio 这里的队列索引使用了比容量所需位数更宽的整数类型（`UnsignedShort` 为 16 或 32 位，容量固定为 256）。具体做法：

- 索引 `head` 和 `tail` 一直通过 `wrapping_add` 递增，从不主动对容量取模，任由它们自然溢出。

- 将逻辑索引映射到缓冲区物理下标时，才用位与掩码求模：`idx = tail as usize & MASK`（`MASK = N - 1`）。

- 计算队列长度时，直接用 `tail.wrapping_sub(head)`。因为两个宽整数即使绕过了最大值，在有符号减法（或无符号回绕减法）视角下，它们的差值始终等于两者在环形顺序上的距离，也就是队列中元素的数量。

这样一来：

- 空队列：`tail == head` → 差值为 `0`。

- 满队列：`tail - head` 正好等于 `CAPACITY`（如 256），因为从头到尾总共写入了 `CAPACITY` 个元素，而索引本身足够宽，差值可以直接等于容量值。

- 无需浪费槽位，也无需取模或额外的标志，就可以直接、无歧义地获得队列长度，并区分空/满。

这就是“由于索引超宽，直接差值即为队列长度，无模运算”的含义——借由让索引拥有远大于容量的取值范围，把取模操作推迟到实际寻址的那一刻，长度计算退化为一次简单的减法，从而简化并发控制并提高效率。

1. 初始状态

```text
head = 0
tail = 0
长度 = tail.wrapping_sub(head) = 0  ✅ 空队列
物理写入位置 = 0 & 255 = 0
```

2. 推入 1 个任务

```text
head = 0
tail = 1
长度 = tail.wrapping_sub(head) = 1
物理写入位置 = 1 & 255 = 1
```

3. 推满队列（256 个任务）

```text
head = 0
tail = 256
长度 = tail.wrapping_sub(head) = 256  ✅ 满队列
物理写入位置 = 256 & 255 = 0
```

4. 弹出 10 个任务后

```text
head = 10
tail = 256
长度 = tail.wrapping_sub(head) = 246
物理写入位置 = 256 & 255 = 0
```

5. 再推入 10 个任务（达到满）

```text
head = 10
tail = 266
长度 = tail.wrapping_sub(head) = 256  ✅ 满队列
物理写入位置 = 266 & 255 = 10
```
