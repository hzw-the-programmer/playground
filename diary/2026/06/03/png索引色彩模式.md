# 一、三大组成要素

索引图由三部分配合生效：

1. IHDR.color_type = 3：标记本图是索引模式
2. PLTE 关键块（必须存在）：全局调色板，存储全部真实 RGB 颜色
3. IDAT：像素数据只存索引编号，不是 RGB 值
4. 可选：tRNS：给部分调色板颜色设置全透明

## 1. `PLTE` 调色板块规则

- Data 由若干 [R, G, B] 三元组组成，每组 3 字节；
- 长度：N × 3，1 ≤ N ≤ 256（最多 256 色，所以叫 PNG8）；
- 下标从 0 ~ N-1，像素的值就是下标。

例：PLTE [5] = [255,0,0] → 像素值 = 5 就显示红色。

## 2. `bit_depth` 决定索引占用 bit、最大颜色数

bit_depth	单个索引占用 bit	索引取值范围	最大可用颜色
1	1 bit	0 / 1	2 色（黑白）
2	2 bit	0~3	4 色
4	4 bit	0~15	16 色
8	8 bit	0~255	256 色 (PNG8)

存储优化：一行像素打包位，例如 bit=4，1 字节存 2 个索引。

# 二、像素数据存储逻辑

普通 RGB：1 像素 = 3 字节 (RGB)
索引 PNG8 (bit8)：1 像素 = 1 字节（仅索引），体积直接压缩到原图 1/3。

```plaintext
像素值 = idx → 查表PLTE[idx] → R/G/B
```

示例：

`PLTE`:

```plaintext
0:黑(0,0,0)
1:红(255,0,0)
2:白(255,255,255)
```

像素流：[1,0,2] → 像素依次：红→黑→白

# 三、`tRNS` 透明规则（索引色独有透明逻辑）

`tRNS` 是可选辅助块，专门给索引色做透明：

- `tRNS` 数据长度 = 调色板有效颜色数量 N
- tRNS[n] = 0：PLTE 第 n 号颜色完全透明
- tRNS[n] != 0：不透明

关键限制：

索引色只能整色全透明，不支持单个像素半透明（没有 Alpha 通道），无法做渐变阴影。
想要半透明必须换成 RGBA（color_type=6 PNG32）。

# 四、解码全流程（png crate 源码逻辑）

- 读 `IHDR` → color_type=3，判定索引图
- 读取 `PLTE` → 在内存保存 RGB 调色板数组
- 读取 `tRNS`（存在则生成透明映射表）
- `IDAT` 解压 + 逆滤波 → 原始索引字节流
- 遍历每个索引，查表转为 RGB 像素 → 上层 image::Rgb

# 五、编码流程（生成 PNG8）

- 原图 RGB 像素集合做色彩量化 (color_quant)：把成千上万种颜色缩减≤256 色，生成 `PLTE`
- 每个原始 RGB 查找在调色板中的下标，生成索引数据流
- 行滤波 + Deflate 压缩存入 IDAT
- 需要透明：把指定颜色下标写入 tRNS

Rust crate：color_quant 专门做调色板量化。

# 六、优缺点

✅ 优点

- 体积极小：256 色图相比 PNG24 体积≈1/3~1/8
- 兼容透明 (tRNS)，替代 GIF，无专利
- 移动端图标、ICO、小贴图首选

❌ 缺点

- 最多 256 色，复杂照片色彩断层严重
- 无逐像素 Alpha，不能半透明渐变

# 七、常见误区

- ❌ PNG8 一定透明：不一定，没有 `tRNS` 就完全不透明
- ❌ PNG8=8bit 色深：bit=1/2/4 也属于索引，只有 bit8 才是标准 PNG8
- ❌ 索引能半透明：不能，半透明只能 RGBA

# 八、和其它模式对比

格式	存储内容	透明
PNG8 (索引 type3)	索引 + `PLTE`	指定色全透，无半透
PNG24(RGB type2)	R/G/B	`tRNS` 单色全透
PNG32(RGBA type6)	R/G/B/A	逐像素 0~255 半透明

# 九、Rust 代码极简示意（逻辑伪代码）

```rust
// 索引转RGB
let plte = &[[0,0,0],[255,0,0],[255,255,255]];
let trns = &[0,255,255]; // 0号色透明
let indexes = [1,0,2];

for &idx in &indexes {
    let rgb = plte[idx];
    let alpha = if trns[idx]==0 {0} else {255};
}
```
