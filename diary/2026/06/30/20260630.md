# SQLx 整体数据结构分层设计详解（0.7.x）

SQLx 采用**分层模块化**结构，分为：**驱动抽象层、核心通用层、查询构建层、宏编译层、连接池层、行 / 类型映射层**，全程基于 Rust trait 做驱动多数据库兼容，泛型零成本抽象。

## 总架构分层

1. **Trait 抽象底座：Database / Connection / Row / Statement**

统一多数据库接口（Postgres/MySQL/SQLite），屏蔽各数据库协议差异；

2. **核心通用数据结构：Query、QueryAs、Arguments、Either、Statement**

承载 SQL、参数、预编译句柄，业务直接操作；

3. **行与类型映射：Row、FromRow、Value、TypeInfo**

数据库二进制 ↔ Rust 类型转换核心；

4. **连接与池：Pool、PoolConnection、Transaction**

会话、事务、连接复用管理；

5. **宏编译层：sqlx-macros 内部元数据结构 QueryMetadata**

query! 编译期解析 SQL 用到的缓存结构；

6. **离线缓存：sqlx-data.json 对应离线元数据结构体**

---

## 一、底层抽象 Trait（整个库的设计基石）

所有数据库驱动都实现同一套 trait，上层代码无感知切换数据库。

### 1\. `Database` trait

零标记 trait，绑定整套关联类型，是泛型约束的核心：

```rust
trait Database {
    type Connection: Connection<Self>;
    type Row: Row<Self>;
    type Statement: Statement<Self>;
    type Value: Value<Self>;
    type Arguments: Arguments<Self>;
    type Error: DatabaseError;
}
```

以 Postgres 为例：

Postgres 实现 `Database`，关联类型：

- `Connection` = `PgConnection`

- `Row` = `PgRow`

- `Statement` = `PgStatement`（预编译语句句柄）

作用：`Query<'_, DB, A>` 中的 `DB` 就是该 trait，编译期区分数据库语法、占位符、类型映射。

### 2\. `Statement<DB: Database>`

对应数据库原生 **Prepared Statement（预备语句）** 句柄。

存储：

- 预编译后的服务端语句 ID；

- 参数类型列表、返回列元数据；

生命周期绑定连接，连接销毁则 Statement 失效。

两种创建途径：

- 第一次执行 `Query`：内部发送 `PREPARE`，生成 Statement 存入连接缓存；

- `query!` 宏在线模式：临时 `PREPARE` 获取元数据，用完立即 `DEALLOCATE`。

### 3\. `Arguments<DB>`

参数容器，存储绑定给 SQL 的所有参数二进制值：

- 动态变长存储，支持链式 .bind()；

- 内部统一转为数据库协议二进制，避免字符串拼接防注入；

- 校验参数数量与 SQL $1/$2 匹配，提前返回错误。

### 4\. `Row<DB>`

单条查询结果行，存储列名、二进制值数组：

- `try_get<T>(col_name/index)`：类型安全取值；

- 驱动实现：`PgRow` / `MySqlRow`，内部持有协议原始数据包。

### 5\. `FromRow<'r, R: Row>`

行映射 trait，自定义结构体 / 宏生成匿名结构体都实现它：

```rust
trait FromRow<'r, R: Row> {
    fn from_row(row: &'r R) -> Result<Self, Error>;
}
```

`query_as!(T)`、`#[derive(sqlx::FromRow)]` 本质都是实现该 trait。

## 二、核心业务结构：`Query` 全链路设计

```rust
#[must_use = "query must be executed to affect database"]
pub struct Query<'q, DB: Database, A> {
    pub(crate) statement: Either<SqlStr, &'q DB::Statement>,
    pub(crate) arguments: Option<Result<A, BoxDynError>>,
    pub(crate) database: PhantomData<DB>,
    pub(crate) persistent: bool,
}
```

### 1\. 核心设计思路：延迟预编译 + 双模式存储

`statement: Either<SqlStr, &'q DB::Statement>`

`Either<L, R>` 枚举二选一：

1. `Left(SqlStr)`：原始 SQL 字符串（首次执行）
   
   执行时自动发送 `PREPARE`，生成 `Statement` 缓存到连接；

2. `Right(&DB::Statement)`：已预编译句柄（重复执行）
   
   跳过解析，直接 `EXECUTE`，性能更高。

`arguments: Option<Result<A, BoxDynError>>`

- `A`：参数元组 `()` / `(T1,)` / `(T1,T2)`；

- `None`：未绑定任何参数；

- `Some(Ok(A))`：参数绑定成功；

- `Some(Err(e))`：绑定阶段类型不匹配，提前缓存错误，执行时直接返回，不访问数据库。

`persistent: bool` **缓存策略开关**

- `true`：业务普通查询，连接归还池不销毁 Statement，复用预编译计划；

- `false`：宏编译期临时探测语句，用完立即 DEALLOCATE，不占用连接缓存。

`PhantomData<DB>`

零开销类型标记，仅用于编译期区分数据库驱动，无运行时内存占用。

### 2\. `Query` 两大变体（强类型 / 动态）

1. **动态 Query**：`sqlx::query()` → `Query<'q, DB, A>`

   输出 `Row`，无编译期校验，运行时解析类型；

2. **强类型 Query**：`sqlx::__query_with_result<O, DB>`（`query!` 宏生成）

   泛型 `O: FromRow`，自动生成匿名结构体，编译期校验 SQL、字段、类型。

### 3\. `Query` 生命周期流转

1. 构造：仅保存 SQL/Statement 引用，**不访问数据库**；

2. `.bind()`：追加参数，修改 `arguments`，返回新 `Query`；

3. `.fetch_one`/`fetch_all`/`execute()`：消耗 Query，发起 IO：

   - 校验提前缓存的参数错误；
   
   - 区分字符串 / 预编译句柄；
   
   - 发送 EXECUTE 并解析返回行。

### 4\. 配套衍生结构

- `QueryAs<'q, DB, O, A>`：封装强类型映射逻辑；

- `ExecuteResult`（`PgQueryResult`）：DML 执行结果，仅保存受影响行数。

## 三、行与类型转换体系（数据库 ↔ Rust）

### 1\. `Value<DB>`

数据库原始字段二进制封装，驱动实现：

- `PgValue`：存储 Postgres 二进制、OID 类型、是否 NULL；

   提供 `try_into<T>` 做类型转换。

### 2\. `TypeInfo`

存储数据库字段元数据：

- 类型名称、OID、是否可空；

  `query!` 宏在线模式通过 PREPARE 拉取该结构，用于生成 Rust 类型。

### 3\. 类型映射规则结构

内部维护映射表（编译期查表）：

- PG `timestamptz` → `time::OffsetDateTime` / `chrono::DateTime<FixedOffset>`；

- `BIGINT` → `i64`，`TEXT` → `String`；

可空字段自动包装 `Option<T>`。

### 4\. `FromRow` 派生结构

```rust
#[derive(sqlx::FromRow)]
struct User {
    id: i64,
    name: String,
}
```

过程宏自动生成 `FromRow` impl，内部调用 `row.try_get("字段名")`。
`query!` 自动生成匿名结构体，同样实现 `FromRow`。

## 四、连接、事务、连接池结构

### 1\. `Connection<DB>`

单条数据库会话，持有：

- TCP/TLS 连接流；

- 当前会话缓存的 `Statement` 预备语句；

- 事务状态标记（是否在事务中）。

### 2\. `Transaction<'c, DB>`

包装 `&'c mut Connection`，新增事务状态：

- `commit()` / `rollback()` 控制事务生命周期；

- 内部复用连接的 `Statement` 缓存。

### 3\. `Pool<DB>` 连接池

核心字段：

- 空闲连接队列；

- 最大 / 最小连接数配置；

- 后台保活、新建连接任务；

- 对外仅暴露 `&Pool` 共享引用（`Arc` 内部封装），多线程安全。

### 4\. `PoolConnection`

RAII 包装连接，离开作用域自动归还池，不会直接暴露原始 Connection。

## 五、`sqlx-macros` 编译期元数据结构（离线缓存核心）

`QueryMetadata`（宏核心数据结构）

在线 PREPARE / 离线 `sqlx-data.json` 统一存储该结构体：

```rust
struct QueryMetadata {
    // SQL 原文哈希，用于离线缓存匹配
    sql_hash: [u8; 16],
    // $1 $2... 参数类型列表
    params: Vec<TypeInfo>,
    // SELECT / RETURNING 返回列元数据
    columns: Vec<ColumnMeta>,
}

struct ColumnMeta {
    name: String,
    type_info: TypeInfo,
    nullable: bool,
}
```

**两种加载路径**

1. **在线模式**：编译时连接数据库，执行 `PREPARE`，构造 `QueryMetadata`；

2. **离线模式**：`sqlx prepare` 提前批量生成 `sqlx-data.json`，编译时读取 JSON 反序列化为该结构，无需数据库。

**宏生成产物**

根据 `QueryMetadata` 自动生成：

1. 匿名结构体（匹配 columns 名称、类型、Option）；

2. 结构体 `FromRow` 实现；

3. 调用 `__query_with_result` 强类型 `Query` 构造代码。

## 六、辅助通用工具结构

### 1\. `Either<L, R>`

二选一枚举，统一「字符串 SQL / 预编译 Statement」两种存储模式，广泛用于 Query。

### 2\. `BoxDynError`

类型擦除错误，统一存储绑定参数阶段各种类型错误。

### 3\. `PhantomData<T>`

无大小标记，用于泛型 DB、生命周期约束，零运行时开销。

### 4\. `MapOkOrElse` / `MapOk` / `MapErr`

TryFuture 包装 Future 结构体，内部存储原始 Query Future + 转换闭包，实现链式错误处理。

## 七、整体数据流结构串联（完整链路）

### 1\. 编码阶段（写代码）

```rust
sqlx::query!("SELECT id,name FROM users WHERE id=$1", 1i64)
```

过程宏读取 SQL → 加载 `QueryMetadata` → 生成匿名结构体 + `__query_with_result`。

### 2\. 编译阶段

1. 在线：连接数据库 PREPARE，拉取 `QueryMetadata`；

2. 离线：读取 `sqlx-data.json` 匹配 SQL 哈希获取元数据；

3. 根据元数据生成强类型 `Query` 代码。

### 3\. 运行时构建 `Query`

```rust
Query {
    statement: Either::Left(SqlStr("SELECT ...")),
    arguments: Some(Ok((1i64,))),
    database: PhantomData<Postgres>,
    persistent: true,
}
```

### 4\. 执行 `.fetch_one(pool).await`

1. 从 `Pool` 获取 `PoolConnection` → 拿到 `PgConnection`；

2. 判断 `statement` 是字符串：发送 `PREPARE`，缓存 `PgStatement`；

3. 绑定参数二进制，发送 `EXECUTE`；

4. 服务端返回数据包 → 解析为 `PgRow`；

5. 调用自动生成的 `FromRow::from_row` 转为匿名结构体返回。

### 5\. 连接归还

连接放回池，`persistent=true` 保留 `PgStatement`，下次复用无需 PREPARE。

## 八、设计核心优势总结

1. **强分层抽象，多数据库兼容**

   顶层业务只操作 `Query`/`Row`，底层数据库差异被 `Database`/`Connection` trait 屏蔽；

2. **延迟预编译，性能分层**

   `Query` 创建不碰数据库，重复 SQL 复用 Prepared Statement；

3. **编译期类型安全（宏体系）**

   `QueryMetadata` 预解析 SQL，表 / 字段 / 类型错误编译报错，规避线上 SQL 异常；

4. **零成本泛型抽象**

   `PhantomData`、泛型参数、组合 `Future` 全部编译内联，无堆分配运行时损耗；

5. **RAII 资源安全**

   `Connection`/`Transaction`/`PoolConnection` 生命周期自动管理连接、事务、预编译句柄；

6. **参数与 SQL 完全分离**

   `Arguments` 单独存储二进制参数，天然防 SQL 注入；

7. **在线 / 离线双编译模式**

   `QueryMetadata` 同时支持实时数据库探测与本地 JSON 缓存，适配 CI 无数据库环境。

---

## 九、对比普通 ORM 的结构特色

1. 不维护实体映射表，运行时无反射，全部类型信息编译期确定；

2. Prepared Statement 作为一等公民内置在 `Query` 结构，不是附加功能；

3. 区分动态 `query()` 和静态 `query!` 两套结构，兼顾灵活与安全；

4. 错误分层：绑定阶段错误、SQL 语法错误、行类型转换错误分层捕获。
