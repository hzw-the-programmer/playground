`Arc<T>` 要实现 `Sync`（即 `&Arc<T>` 可以安全地在多个线程间共享），需要 `T: Send + Sync`，原因同样从 `Arc` 的两个能力出发：

# 1. 共享 `&T` 需要 `T: Sync`

多个线程通过共享的 `&Arc<T>` 可以调用 `Deref` 获得 `&T`，这意味着多个线程同时持有 `T` 的共享引用。
要保证这是安全的，`T` 必须支持并发只读访问（或内部同步），即 `T: Sync`。

# 2. 通过共享引用 `clone` 出新的所有权，可能跨线程析构 `T`

这是容易被忽略的一点。`Arc::clone` 的签名是 `fn clone(&self) -> Arc<T>`，它只需要 `&self`（即 `&Arc<T>`）。
因此，多个线程如果持有同一个 `&Arc<T>`，它们都可以调用 `clone` 创建出新的 `Arc<T>` 实例。这些新实例可以被任意移动到其他线程，并在那里被丢弃。当最后一个 `Arc<T>` 析构时，它会在那个线程内执行 `T` 的析构并释放内存。

这等价于 `T` 的所有权可能被转移到任何一个线程。所以 `T` 必须能安全地跨线程发送，即 `T: Send`。

# 完整逻辑链

`Arc<T>: Sync` 意味着：

- 多线程共享 `&Arc<T>`
  + 可从中获取 `&T` → 要求 `T: Sync`
  + 可从中 `clone` 出 `Arc<T>` → 新的 `Arc<T>` 可被发到任何线程并析构 `T` → 要求 `T: Send`

因此，标准库的实现是：

```rust
unsafe impl<T: ?Sized + Sync + Send, A: Allocator + Sync> Sync for Arc<T, A> {}
```

# 一句话总结

`Arc<T>` 要允许通过共享引用安全地克隆出新的所有权并最终在任意线程释放数据，所以既要求 `T` 能跨线程共享（`Sync`），又要求它能跨线程转移所有权（`Send`）。这和 `Rc<T>` 完全相反——`Rc` 的 `clone` 因为非原子计数本身就线程不安全，所以连 `Sync` 的边都沾不上。
