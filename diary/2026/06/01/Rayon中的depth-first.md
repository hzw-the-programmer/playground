在 Rayon 的上下文中，depth-first（深度优先） 和 breadth-first（广度优先） 指的是工作线程执行自己本地任务队列的顺序策略，这直接影响工作窃取（work-stealing）的行为和任务的执行顺序。

# 深度优先（Depth-first，默认）

- 执行顺序：后进先出（LIFO），线程优先执行最新产生的任务。

- 行为：当一段代码通过 join 或 scope 产生两个子任务 A 和 B 时，线程会把一个任务（比如 B）放入本地队列，然后立即开始执行另一个（A）。当 A 完成后，它会从队列尾部弹出最近放入的任务继续执行。这就形成了一个不断“深入”的模式——先处理完最内层分叉出来的任务，再回溯处理之前的兄弟任务。

- 优点：

  + 缓存友好：继续执行刚创建的任务意味着能更好地利用 CPU 缓存，因为数据很可能还是“热”的。

  + 降低窃取频率：新产生的任务往往粒度较小，线程自己快速消化这些细粒度任务，留给其他线程窃取的通常是更早产生的、粒度可能更大的任务，从而减少窃取开销，提高整体效率。

- 典型场景：递归分治算法（如快速排序、归并排序）中，深度优先天然契合“先处理子问题，再合并”的模式。

# 广度优先（Breadth-first）

- 执行顺序：先进先出（FIFO），线程优先执行最早产生的任务。

- 行为：线程以“层”的方式展开工作。当 join 产生子任务时，线程把任务放入队列尾部，但会从队列头部取出最老的任务来执行。这样，它会先处理同一层级上所有分叉出来的任务，再进入下一层级。

- 优点：

  + 更早暴露并行性：广度优先更容易让各个线程很快看到大量可并行执行的任务，有可能更充分地利用多核。在任务依赖少、希望立即扩散并行度的场景中有用。

  + 避免过深的本地栈：深度优先在极端递归下可能导致单一线程的调用栈过深或本地队列长时间持有大量任务，而广度优先能更均衡地“摊开”工作。

- 代价：

  + 缓存不友好：频繁在不同任务上下文间切换，缓存失效率可能上升。

  + 更高的窃取开销：线程自己从队列头部取走的经常是粒度较大的任务，而新产生的细粒度任务留在队列尾部，很容易被其他线程窃取，导致大量小块任务被频繁窃取，增加了同步和调度开销。

# 总结：

breadth_first = false（深度优先）是默认值，适合大多数并行计算场景，因为它结合了良好的缓存局部性和较低的窃取成本。只有当你的算法有明确的“先展开同一层级所有任务”的需求，或者深度优先导致了负载严重不均时，才需要显式设置为 true 使用广度优先。
