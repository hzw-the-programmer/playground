# 日常
- 跟着豆包学加密
  
  ECH（Encrypted Client Hello）
  HPKE（Hybrid Public Key Encryption）
  ECH 是 TLS 扩展（解决 SNI 隐私泄露），HPKE 是底层加密框架（ECH 的加密工具）。
  解决问题：彻底加密 TLS 握手的 ClientHello，隐藏 SNI（真实域名），防止 ISP / 防火墙 / 中间人窥探用户访问目标。
  核心机制（双层 ClientHello）
  Outer ClientHello（明文外层）
  发 “马甲域名”（public_name，如 cloudflare-esni.com）
  含 encrypted_client_hello 扩展（装加密内容）
  Inner ClientHello（加密内层）
  真实 SNI、密码套件、ALPN 等敏感信息
  用 HPKE 加密（服务器 ECH 公钥）
  服务器处理
  能解密 → 用 Inner 完成握手
  不能解密 → 用 Outer 降级握手（不中断）

  关键特性
  替代 ESNI：更安全、兼容更好、解密失败不中断
  配合 DoH：ECH 公钥通过 HTTPS DNS 安全获取
  TLS 1.3 专属：IETF 标准（draft-ietf-tls-esni）

  HPKE：混合公钥加密（通用密码框架）
  RFC 9180，ECH 的官方加密算法。
  解决问题：安全、标准、高效的 “公钥加密小数据”（替代 ECIES 等旧方案）。
  三层架构（KEM + KDF + AEAD）
  KEM（密钥封装）
  用 DH（X25519/secp256r1/FFDHE）生成共享密钥
  输出：enc（临时公钥）+ shared_secret
  KDF（密钥派生）
  HKDF-SHA256 扩为：AEAD key + nonce + exporter_secret
  AEAD（认证加密）
  AES-GCM / ChaCha20-Poly1305 加密数据

  为什么 ECH 选 HPKE？
  标准化：RFC 9180，跨库兼容（ring/aws-lc-rs/rustls）
  安全：强前向保密、防密钥复用、上下文绑定
  简洁：一套 API 覆盖 ECH/MLS/SSH 等场景

  ECH ↔ HPKE 关系（一句话）
  ECH 是 “隐私目标”，HPKE 是实现它的 “加密引擎”。
  工作流程（ECH 使用 HPKE）
  客户端通过 DoH 取服务器 ECHConfig（含 HPKE 公钥）
  构造 Inner ClientHello（真实 SNI）
  HPKE 加密 Inner → 密文 + enc（临时公钥）
  构造 Outer ClientHello（马甲域名）
  发送：Outer + encrypted_client_hello（密文 + enc）
  服务器：HPKE 解密 → 读 Inner → 握手

  极简总结
  HPKE = 现代、安全的公钥加密工具箱
  ECH = TLS 隐私扩展，用 HPKE 加密 SNI

  KEM = Key Encapsulation Mechanism
  用对方的公钥，安全地 “封装” 出一个共享对称密钥；对方用私钥 “解封装” 得到同一个密钥。

  KEM 解决什么问题？
  以前我们做安全通信是这样：
  协商一个随机 AES 密钥
  用对方RSA 公钥加密这个 AES 密钥发给对方
  对方 RSA 解密得到 AES 密钥

  这叫：公钥加密 → 传输会话密钥

  但它有缺点：
  语义安全性不够强
  不适合后量子算法
  格式复杂、容易写错

  KEM 就是把这套流程标准化、现代化、更安全。

  2. KEM 标准流程（只有 2 个 API）
  ① 发送方：封装（Encapsulate）
  '''rust
  let (ciphertext, shared_secret) = kem.encapsulate(recipient_public_key);
  '''

  输入：对方公钥
  输出：
  ciphertext (enc)：要发给对方的 “封装结果”
  shared_secret：本地算出的共享密钥

  ② 接收方：解封装（Decapsulate）
  '''rust
  let shared_secret = kem.decapsulate(recipient_private_key, ciphertext);
  '''
  输入：自己私钥 + 对方发来的 ciphertext
  输出：同一个 shared_secret

  双方得到相同密钥，直接用于 AES-GCM。

  3. KEM 为什么比传统 RSA 加密更好？
  天然语义安全（即使相同输入，每次输出不同）
  一次函数调用完成密钥协商，API 干净
  天然适合后量子密码（ML-KEM 就是 KEM）
  没有 “选择密文攻击” 漏洞
  HPKE、ECH、TLS 1.3 QUIC、MLS 全都用 KEM

  4. 常见 KEM 算法
  ① 传统 ECDH 类 KEM
  X25519
  secp256r1 (P-256)
  secp384r1
  FFDHE

  这些在 TLS 里本质都是 KEM：

  公钥 = 对方 ECDH 公钥
  ciphertext = 自己的 ECDH 公钥
  shared_secret = ECDH 共享密钥（IKM）

  ② 后量子 PQ KEM（NIST 标准）
  ML-KEM（FIPS 203） ← 新一代 TLS 1.3 要用
  抗量子计算机攻击
  未来会替代 ECDH

  5. KEM 在你之前学的东西里的位置
  

  DER 最核心结构：TLV（Type–Length–Value）
  [1字节类型] + [长度] + [实际值]
  PEM 只是 DER 的文本包装！
  PEM = Base64(DER) + 头尾标记行
  -----BEGIN CERTIFICATE-----
  MIICUTCCAfugAwIBAgIBADANBgkqhkiG9w0BAQQFADBX...
  -----END CERTIFICATE-----
  解码 Base64、去掉头尾 → 得到 DER 二进制。

  DER = 证书、公钥、私钥、签名的【标准二进制格式】

  1. DER 解决什么问题？
  X.509 证书是一个复杂结构：
  版本
  序列号
  签发者
  公钥
  签名算法
  签名值
  扩展字段
  这些是结构化数据，不能直接发网络。
  需要一种严格、紧凑、唯一的二进制格式：
  → DER 就是干这个的。
  
  特点：
  二进制，不是文本
  格式唯一（同样数据只能编出一种结果，适合验签）
  紧凑，没有多余字节
  可解析、可验证

  TLS 握手中传输的永远是 DER，不是 PEM。

  0x30 = SEQUENCE（证书最外层）
  0x02 = INTEGER（序列号、版本）
  0x03 = BIT STRING（公钥）
  0x05 = NULL

  [1字节类型] + [长度] + [实际值]

  30 2E            ← SEQUENCE (总长度 46)
   02 01 00      ← INTEGER version = 0
   30 05         ← SEQUENCE 算法标识
      06 03 2B 65 70   ← OID = 1.3.101.110 (X25519)
   04 22         ← OCTET STRING (私钥容器)
      04 20      ← OCTET STRING (真正的 32 字节 X25519 私钥)
         7F B6 BB 31 ...  ← 32 字节私钥

  DER 06 到底是什么？（超清晰）
  在 DER 编码的 TLV 结构里：
  06 = OBJECT IDENTIFIER（OID，对象标识符）

  1. DER TLV 类型速查表（你最常用的）
  0x02 = INTEGER（整数）
  0x03 = BIT STRING（比特串）
  0x04 = OCTET STRING（字节串）
  0x05 = NULL（空）
  0x06 = OBJECT IDENTIFIER → **就是你问的这个！**
  0x30 = SEQUENCE（序列）
