PostgreSQL 的通信协议是一个**基于消息的、明文（但可加密）的应用层协议**，用于客户端（前端）和服务器（后端）之间的所有交互。它可以通过 **TCP/IP** 或 **Unix 域套接字**进行通信，默认端口为 **5432**。

# 📡 协议通信基础：消息格式

所有通信都由一系列消息构成，每个消息有其固定格式：

- **消息类型**：首个字节标识消息类型（如 'Q' 表示简单查询）。

- **消息长度**：接下来的**4个字节**指明消息内容的长度（包含自身，但不包含类型字节）。

- **消息内容**：剩余部分，格式由消息类型决定。

> **注意**：客户端发送的第一个“启动消息”是唯一的例外，它没有**初始消息类型字节**。协议还支持**文本**和**二进制**两种数据格式。

# 🤝 会话生命周期

一个完整的数据库会话遵循以下流程：

## 1. 🚀 启动与认证阶段

这是建立连接的第一步。

- **发起连接**：客户端发起TCP连接并发送**启动消息**，包含用户名、数据库名和协议版本。

- **协商版本**：服务器检查协议版本。若不支持，会拒绝连接或通过 `NegotiateProtocolVersion` 消息协商一个都支持的版本。

- **身份认证**：服务器根据 `pg_hba.conf` 配置决定认证方式，并发送认证请求。主要方式包括：

  + **明文密码**：服务器发 `AuthenticationCleartextPassword`，客户端回复明文密码。

  + **MD5加密**：服务器发 `AuthenticationMD5Password` 和一个**盐**（Salt），客户端用MD5加密密码和盐后回复。**（注意：此方式已被弃用）**

  + **SASL (推荐)**：现代、安全的认证机制，最常用的是 **SCRAM-SHA-256**。服务器列出支持的SASL机制，客户端选择并开始多轮“挑战-响应”交换。

  + **GSSAPI / SSPI**：主要用于Windows环境的企业级认证。

  + **Trust（免密）**：仅适用于可信网络环境。

- **认证完成**：认证成功后，服务器发送 `AuthenticationOk` 消息，会话进入正常操作阶段。

## 2. ⚙️ 正常操作阶段：两种查询协议

建立连接后，客户端可以通过以下两种协议执行SQL命令。

- **简单查询协议**

适用于执行一次性、无参数的SQL命令。

1. 客户端发送一个类型为 `'Q'` 的消息，内容为文本SQL查询字符串。

2. 服务器立即解析、规划并执行该查询。

3. 服务器返回结果，最后以一个 `CommandComplete` 消息结束。

- **扩展查询协议**

适用于需要**防SQL注入、高性能重复执行、流式传输**等场景。它将执行拆分为三个阶段：

1. **解析 (Parse)**：客户端发送类型为 `'P'` 的查询（可含占位符 `$1`, `$2`）。服务器解析、分析并创建一个**预备语句（Prepared Statement）**。

2. **绑定 (Bind)**：客户端发送类型为 `'B'` 的消息，为预备语句的占位符提供具体参数值，服务器创建一个**门户（Portal）**。

3. **执行 (Execute)**：客户端发送类型为 `'E'` 的消息执行门户。对于 SELECT 等查询，可指定每次获取的行数，实现**游标式流式传输**。

此过程中，预备语句和门户都可被命名和复用，提升性能。

## 3. 🛑 终止阶段

会话结束通常由客户端发起，发送类型为 `'X'` 的**终止消息（Terminate）**。服务器收到后会关闭连接。若连接意外中断，服务器也会回滚未完成的事务。

# 💡 其他重要特性

- **COPY 协议**：用于高效批量导入/导出数据，有 `COPY FROM`（将数据流写入表）和 `COPY TO`（将查询结果流式传出）两种模式。

- **异步操作**：支持**通知响应（NOTIFY）** 和**命令取消（Cancel Request）** 等机制。

- **管道模式 (Pipelining)**：允许客户端在未收到前一个请求的响应时，连续发送多个命令，能显著降低网络延迟的影响。

# 📜 协议版本

- **当前主流版本**：3.0（自PostgreSQL 7.4起）。

- **最新版本**：3.2（自PostgreSQL 18起），与3.0版本向后兼容。
